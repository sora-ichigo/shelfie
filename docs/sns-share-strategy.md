# SNS シェア戦略

> **ステータス**: 意思決定済み
> **最終更新**: 2026-02-04

---

## 背景と目的

Phase 1 の Go-to-Market 戦略として、「モダンでいけてるアプリ」というブランディングで ASO だけでなく SNS での露出を図りたい。読んだ本をシェアすることで、オーガニックな認知拡大と新規ユーザー獲得のループを作る。

## 議論の経緯

### 初期案: アプリ内シェア → Web 版 URL 発行 → SNS シェア

Audible や Spotify のように、アプリからシェアすると Web 版の URL が発行され、SNS 経由で流入した人が Web で体験できる仕組みを検討した。

### 初期案に対する懸念

1. **PMF 前にグロース施策を作るリスク** — コア体験（登録・検索・管理）の磨き込みにリソースを集中すべき Phase 1 で、Web 版の開発・運用コストが分散する
2. **Audible / Spotify との構造的な違い** — これらはブランド認知がある前提で機能する。認知ゼロの Shelfie では Web リンクのクリック率が低く、ファネルも深い（リンク → Web → 興味 → ストア → インストール）
3. **投資対効果** — Flutter Web or 別スタックでの Web 版構築は、Phase 1 の開発規模に対して過大

### 決定: 画像シェアから始める

Web 版ではなく、画像生成によるシェアを Phase 1 の施策とする。

## シェアの分類と実装フェーズ

シェアには「コンテンツシェア」と「プロダクトシェア」の2種類がある。

| 種類 | 内容 | 動機 | フェーズ |
|---|---|---|---|
| **コンテンツシェア** | 読了した本 | 「この本よかった」という自己表現 | **Phase 1** |
| **プロダクトシェア** | ブックリスト、アカウント（本棚全体） | 「このアプリ / 本棚いいよ」という布教 | Phase 2〜 |

コンテンツシェアでブランド認知が溜まった上で、プロダクトシェアが効き始めるという順番。

## Phase 1: 読了した本の画像シェア

### 概要

読了時に「読了カード」（本の表紙 + 星評価 + 一言メモ等）を画像として生成し、Instagram Stories / X 等に直接シェアできる機能。

### この方式を選んだ理由

- 読了直後が最もシェア動機が高い瞬間である
- 1冊単位なので画像デザインがシンプルで済む
- 受け手にとっても「この本気になる」と行動につながりやすい
- Web 版不要で開発コストが圧倒的に低い
- **画像に Shelfie のロゴ・ブランドが入る** → ブランド認知の種まき

### 段階的な拡張パス

```
Step 1（Phase 1）: 画像シェア
  読了カードを画像生成し、SNS に直接シェア

Step 2（Phase 1 後半〜Phase 2）: OGP 付き Web ページ
  シェアリンクをタップすると OGP 付きの LP レベルのページが開く
  本の情報 +「Shelfie で管理する」の CTA

Step 3（Phase 2〜3）: フル Web 体験
  他人の本棚を閲覧できる Web 版
  ソーシャル機能と連動
```

## 必要なタスク

この機能の実現には、実装だけでなく以下の領域のタスクが発生する。

### 要求・要件定義

- シェア画像に含める情報の定義（表紙、タイトル、著者、星評価、メモ、ロゴ等）
- 対応する SNS プラットフォームの決定（X, Instagram Stories, LINE 等）
- シェア導線の UX フロー（どの画面から、どのタイミングで）

### デザイン

- 読了カードのビジュアルデザイン（ブランドガイドラインとの整合）
- 複数テンプレート or 単一デザインの判断
- ダークモード / ライトモード対応
- 各 SNS プラットフォームの画像サイズ・比率への対応

### 実装

- 画像生成ロジック（Flutter での画像レンダリング）
- SNS シェア連携（Share API / プラットフォーム別の対応）
- シェアフローの UI 実装
