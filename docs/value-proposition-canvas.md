## Customer Profile（顧客プロファイル）

### Customer Jobs（顧客が達成したいこと）

**機能的ジョブ**

- 自分が持っている本・読んだ本を把握したい
- 読みたい本を忘れないように記録しておきたい
- 今読んでいる本の進捗を管理したい
- SNS で見かけて気になった本をすぐに登録したい
- テーマやジャンルごとに本を整理したい

**感情的ジョブ**

- 「あの本なんだっけ」という不安をなくし、記録した安心感を得たい
- 読了数やステータスの変化を通じて、読書を続けるモチベーションを保ちたい
- 次に何を読むか迷ったときに、自分の積読リストから選べる状態でいたい

> **重要な区別 — 「ビジュアル本棚」と「モダン UX」は別物**
>
> **ビジュアル本棚訴求 = ユーザーニーズとしては弱い。** 「自分だけの本棚を眺めて満足」は一見キャッチーだが、実際のユーザー行動としては定着しない。ブックミー等、この軸で差別化を試みたアプリは成功しなかった。「本棚の見た目」自体はユーザーがアプリを使い続ける理由にならない。ユーザーの本質的な欲求は「記録して把握できている安心感」と「次のアクションにつながる実用性」にある。
>
> **モダン UX = 初期の最大の差別化要因であり、ブランドそのもの。** 一方で、パフォーマンス（画面遷移の速さ、滑らかなアニメーション）、洗練されたビジュアルデザイン（ダークモード、余白の使い方、タイポグラフィ）、直感的なインタラクション（ジェスチャー操作、ハプティックフィードバック）は、実用機能と一体になってアプリの品質を形成する。これは「見た目がきれい」という装飾の話ではなく、**「使っていて気持ちいい」=「もう古いアプリには戻れない」** という体験のスイッチングコストを作る戦略。
>
> 国内の競合アプリが軒並み UI/UX の刷新に失敗している現状で、**モダン UX は Phase 1 における最大の乗り換え動機**になる。これは「キレイな本棚を作りたい」というニッチな欲求とは根本的に異なり、「イケてるアプリを使いたい」「もっさりしたアプリはストレス」というはるかに広いユーザー心理に刺さる。
>
> つまり：
> - ~~ビジュアル本棚~~ → プロダクトの存在意義（Why）にはならない
> - **モダン UX** → 初期の差別化（How）として極めて有効。ブランド認知にも直結する

**社会的ジョブ**

- 読んだ本やおすすめを友人・フォロワーにシェアしたい（将来）
- 「SNS で話題の本をちゃんと読んでる自分」を自然に表現したい（将来）

### Pains（ペイン）

**本の管理に関するペイン**

- 「あの本読んだっけ？」と購入済み・既読の本を忘れてしまう
- 積読が増えるが、何が積まれているか把握できない
- SNS で気になった本を見つけても、後で思い出せない

**既存アプリに対するペイン**

- **UI/UX が古い・もっさりしている** — 主要アプリ（ブクログ・読書メーター）は10年以上前からのサービスで、画面遷移が遅い、デザインが時代遅れ
- **機能過多で取っつきにくい** — コミュニティ・談話室・レビューSNS 等の機能が多すぎて、単に本を記録したいだけの人には重い
- **AI・最新技術への対応が遅い** — 海外では StoryGraph が AI レコメンデーション・ムードベース検索を実装している一方、国内主要アプリは AI 活用がほぼゼロ
- **開発が停滞気味** — 長期運営の国内アプリは大きな機能更新が少なく、モダンな UX トレンド（ダークモード、アニメーション、ジェスチャー操作等）に追従していない
- **プラットフォーム制約** — ビブリアは iOS 専用、読書メーターはデータエクスポート不可など、各アプリに構造的制約がある

---

### Gains（ゲイン）

- すべての蔵書をいつでもどこでも確認できる
- バーコードをかざすだけで本を登録できる手軽さ
- 読書状況（積読 / 読書中 / 読了 / 読まない）が一目でわかる
- お気に入りリストで本を自由にグルーピングできる
- 読了した本に評価やメモを残せる
- 使っていて心地よい、モダンで洗練された操作体験

## 既存の代替手段と課題

### 国内主要アプリ

| アプリ | ユーザー規模 | 強み | 課題・ペイン |
|---|---|---|---|
| **ブクログ** | 100万人+（ブックオフグループ） | 管理機能が充実、タグ分類、新刊ニュース、CSV エクスポート | UI が古い、SNS 機能弱い、ソート機能不足、Web サービス発の設計 |
| **読書メーター** | 国内最大級（角川/ドワンゴグループ） | コミュニティが活発、1,800万件超のレビュー蓄積 | 画面遷移が遅い、5段階評価なし、データエクスポート不可、並べ替え不可 |
| **ビブリア** | - | シンプルで見やすい、ソート充実 | iOS 専用、カテゴリ分類不可、SNS 機能なし |
| **Readee**（楽天） | - | 楽天購入履歴と自動連携 | 楽天エコシステム依存、機能が限定的 |
| **ブックミー** | - | ビジュアル本棚に特化 | 「本棚を眺める」訴求だけでは定着せず、実用性が不足 |

**共通する構造的課題**:
- いずれも2010年代前半までに確立されたサービスで、UI/UX のモダナイズが遅れている
- AI・機械学習の活用がほぼない（レコメンデーション、自動分類、読書インサイト等）
- 開発リソースが限られ、大きな機能進化が停滞している

### 海外アプリ

| アプリ | ユーザー規模 | 強み | 日本市場での課題 |
|---|---|---|---|
| **Goodreads**（Amazon） | 約1.5億人 | 世界最大の読書 SNS、圧倒的な書誌データ | 日本語書籍のカバレッジが低い、UI が古い、開発停滞 |
| **StoryGraph** | 約380万人 | AI レコメンデーション、ムード検索、詳細な統計 | 日本語未対応、日本の書籍データが不足 |
| **Literal Club** | - | モダン UI、人ベースのレコメンド、オープン API | 日本語未対応、日本のユーザーベースなし |

**海外アプリの共通課題**: 技術的には先進的だが、日本語書籍・楽天/Amazon.co.jp との連携がなく、日本市場では実用的でない

## 3C 分析

### Customer（市場・顧客）

**市場環境**

- 日本の出版市場（紙＋電子）は2025年で約1兆5,462億円（前年比 ▲1.6%、4年連続マイナス）
- 紙の出版は9,647億円（▲4.1%）で縮小傾向だが、Z世代の73%は紙の本を好む
- 電子出版は5,815億円（+2.7%）で伸長するも、成長率は鈍化
- 電子書籍市場は2029年度に8,000億円弱まで成長見込み（年平均成長率 3.1%）

**ユーザー像**

| セグメント | 特徴 | 読書頻度 | 現在の代替手段 |
|---|---|---|---|
| 既存アプリ不満層 | ブクログ/読書メーターを使っているが UI に不満 | 月1冊以上 | ブクログ、読書メーター |
| 記録していない読書層 | 本は読むが記録していない、管理が面倒 | 月1〜3冊 | なし、メモアプリ、SNS投稿 |
| 若年ライト読書層（10代〜20代） | SNS で本を発見、年に数冊読む | 年数冊〜月1冊 | SNS のブックマーク、なし |
| 読書始めたい層 | 読書に興味はあるが何を読むか分からない | ほぼゼロ | SNS、書店 |

**Z 世代の読書行動データ**（マイナビ 2025年調査 / MERY調査）

- Z 世代社会人の38%が読書習慣あり（62%はなし）
- 読む人の73%は紙の本を好み、購入場所は書店が7割超
- 読みたい本を見つけるきっかけ: **SNS が46.2%で最多**
- 月あたりの読書量ボリュームゾーン: 1〜3冊
- 月あたりの書籍費: 500〜3,000円
- 人気ジャンル: 文学・小説 63.5%、経済・ビジネス 34.3%

### Competitor（競合）

**国内競合の構造的弱点**

| 競合 | 運営母体 | 開発体制 | AI 対応 | 弱点 |
|---|---|---|---|---|
| ブクログ | ブックオフグループ | 小規模・維持運用中心 | なし | 書店流通企業が母体、テック投資の優先度が低い |
| 読書メーター | 角川/ドワンゴ（トリスタ） | 中規模 | なし | 大企業傘下だが本サービスへの注力は限定的 |
| ビブリア | 個人開発 | 1名 | なし | iOS のみ、開発リソースに限界 |
| Goodreads | Amazon | ほぼ凍結状態 | なし | Amazon の戦略的優先度が低く、長年放置されている |

**海外新興の動き**

- StoryGraph: AI レコメンデーションで急成長（380万ユーザー）。ただし日本語圏は未対応
- Literal Club: モダン UI で注目されるも、スケールはまだ小さい
- Hardcover / Basmo: ニッチプレイヤー、日本進出の動きなし

### Company（自社）

**強み**

- モダンな技術スタック（Flutter, GraphQL, PostgreSQL）で高速に機能開発できる
- 日本語書籍に強い検索基盤（楽天ブックス + Google Books 横断検索）
- iOS / Android 同時対応のクロスプラットフォーム開発
- AI 機能を追加できる拡張性のあるアーキテクチャ
- 意思決定が速い小規模チーム

**弱み**

- ユーザーベースがゼロからのスタート（ネットワーク効果がない）
- ブランド認知なし
- 書誌データは外部 API 依存（楽天・Google）
- マネタイズモデルが未確定

## Fit（フィット分析）

### コアバリュー

> **「あなたの本棚を、手のひらに」**
>
> 既存の読書アプリが抱える「古い・重い・多すぎる」を解消し、
> 本を読むすべての人にとっての must-have な本棚体験を提供する。
>
> - **Why（存在意義）**: 気になった本を即記録し、自分の読書状況が常に把握できている安心感
> - **How（差別化）**: 国内アプリで唯一の「モダン UX × 日本語書籍 × 最新テクノロジー」。使い心地そのものが競合との壁になる
> - **What（機能）**: バーコードスキャン、ステータス管理、ブックリスト、評価・メモ → AI レコメンド（将来）

### ターゲットユーザー

**初期フォーカス（ローンチ〜PMF）**

- 年間5冊以上本を読み、蔵書を記録・管理したいと感じている人
- 既存の読書アプリ（ブクログ・読書メーター等）を使ったが、UI の古さや使いにくさに不満を感じている人
- シンプルで洗練されたツールを好む人

**将来ビジョン**

- 10代〜20代を含む、本を読む（または読みたい）すべての人の must-have
- 「読書家」を自称しない層 — SNS で話題の本が気になる、年に数冊は読む、これから読書を始めたい人
- Z 世代の読書行動（SNS で本を発見、紙の本を好む）にフィットするプロダクト

**広がりの仮説**: 「狭く深く」→「広く深く」の段階的拡大

```
Phase 1: 既存読書アプリに不満を持つ管理意識の高い読書家
         → シンプル × モダン UI で乗り換え需要を獲得
Phase 2: 本は読むが記録していない層
         → バーコードスキャンの手軽さと心地よい UX で習慣化を促進
Phase 3: 本を読みたいと思っている・SNS で本に触れている若年層
         → AI レコメンデーション、ソーシャル機能で「次の一冊」との出会いを提供
```

### 競合との差別化ポイント

| 観点 | 国内既存アプリ | 海外先進アプリ | **Shelfie** |
|---|---|---|---|
| UI/UX | レガシー、画面遷移が遅い | モダンだが日本語非対応 | **モダン × 日本語ネイティブ** |
| 日本語書籍 | 強い（楽天連携等） | 弱い | **楽天 + Google 横断検索** |
| AI 活用 | ほぼなし | StoryGraph がリード | **今後の最大の差別化領域** |
| 開発速度 | 停滞気味 | 活発 | **モダンスタックで高速に進化** |
| ターゲット | 読書家コミュニティ | 英語圏の読書家 | **日本の全読書層** |

### Shelfie が勝てる理由

1. **国内既存アプリへの不満が顕在化している** — UI が古い、遅い、重いという声は明確に存在する
2. **海外の先進アプリが日本市場に入れていない** — 日本語書籍データと検索 API の壁がある
3. **「モダン UI × 日本語書籍 × AI」の組み合わせが空白地帯** — この交差点にいるプレイヤーがいない
4. **技術的に正しいスタックを選択済み** — Flutter, GraphQL, モダン API 設計で継続的な進化が可能

## ロードマップ仮説

### Phase 1: コア体験の磨き込み（現在〜PMF）

**目標**: 既存アプリからの乗り換えユーザーが「これでいい」ではなく「これがいい」と感じる体験

| 注力領域 | 具体的な施策 |
|---|---|
| 登録体験の最適化 | バーコードスキャンの精度・速度向上、検索結果の品質改善 |
| 本棚の実用性強化 | 読書統計（月別冊数、累計推移）、ソート・フィルタの拡充 |
| データ移行 | ブクログ・読書メーターからの CSV / データインポート機能 |
| 安定性・パフォーマンス | オフライン対応強化、起動速度・画面遷移の最適化 |

### Phase 2: 習慣化と定着（PMF 後〜グロース初期）

**目標**: 「本は読むが記録していない」層が自然に使い始め、続けられる仕組み

| 注力領域 | 具体的な施策 |
|---|---|
| 読書統計・インサイト | 年間読了数、ジャンル分布、読書ペース可視化 |
| 通知・リマインド | 積読リマインダー、読書ストリーク |
| シェア機能（軽量） | 読了した本を SNS にワンタップシェア、本棚のスクリーンショットシェア |
| ウィジェット | ホーム画面に「今読んでいる本」「次に読む本」を表示 |

### Phase 3: 発見と出会い（グロース〜スケール）

**目標**: 10代〜20代の「読書家じゃない層」の must-have になる

| 注力領域 | 具体的な施策 |
|---|---|
| AI レコメンデーション | 読書履歴・評価に基づく「あなたへのおすすめ」、ムード/気分ベースの提案 |
| ソーシャル機能 | フォロー、友達の読書アクティビティ、おすすめリストの共有 |
| SNS 連携 | TikTok/Instagram で話題の本のトレンド表示、SNS 投稿からの本登録 |
| AI 読書アシスタント | 「この本どんな話？」「似た本を教えて」など対話型の本探し |

### Phase 4: プラットフォーム化（スケール後）

| 注力領域 | 具体的な施策 |
|---|---|
| 出版社・書店連携 | 新刊情報、書店在庫検索、購入導線 |
| 読書コミュニティ | 読書会、テーマ別ディスカッション |
| API 公開 | サードパーティ連携、ウィジェット開発者向け API |
| Web 版 | ブラウザからもアクセス可能にする |

## 市場規模と売上シミュレーション

### TAM / SAM / SOM

**前提データ**

| 指標 | 数値 | 出典 |
|---|---|---|
| 日本の16歳以上人口 | 約1億800万人 | 総務省 人口推計 2025年 |
| 月1冊以上読書する人の割合 | 37.4%（62.6%が「読まない」） | 文化庁 令和5年度 国語に関する世論調査 |
| 月1冊以上読書する人口 | 約4,040万人 | 上記より算出 |
| 年に数冊は読む人（月1冊未満含む） | 推定50〜55%（約5,400〜5,940万人） | 「読まない」層にも不定期に読む人が含まれる |
| スマートフォン保有率（16歳以上） | 約90% | 総務省 通信利用動向調査 |
| 国内読書管理アプリの推定ユーザー総数 | 500〜800万人 | ブクログ200万+、読書メーター数百万+、その他含む推定 |

**TAM（Total Addressable Market）— 最大市場規模**

日本で年1冊以上本を読む人 × スマートフォン保有者

> **約4,900万人**（5,400万人 × 90%）
>
> 仮にこの全員が月額480円のサブスクに加入した場合の理論上限:
> 4,900万人 × ¥5,760/年 = **約2,822億円/年**

※これは理論上限であり、到達可能な数字ではない。市場のポテンシャルを示す指標。

**SAM（Serviceable Addressable Market）— 提供可能市場**

読書管理アプリに関心を持ちうるユーザー層（定期的に読書し、記録に関心がある or 既存アプリを利用中）

> **約800〜1,200万人**
>
> 算出根拠:
> - 月1冊以上読む4,040万人のうち、記録ニーズがある層を20〜30%と推定
> - 既存アプリユーザー500〜800万人 + 潜在的な新規層
>
> サブスク（課金率5%想定）+ アフィリエイト: **約35〜53億円/年**
>
> | 収益源 | 計算 | 金額 |
> |---|---|---|
> | サブスク | 1,000万人 × 5% × ¥5,760/年 | 28.8億円 |
> | アフィリエイト | 1,000万人 × 10% 購入率 × ¥1,200/件 × 3% 報酬率 × 年6回 | 2.2億円 |
> | **合計** | | **約31億円** |

**SOM（Serviceable Obtainable Market）— 獲得可能市場**

Shelfie が3〜5年で現実的に獲得可能なシェア

> **Phase 3 到達時（3〜5年後）: MAU 50〜200万人**
>
> 算出根拠:
> - 国内読書管理アプリ市場で10〜20%のシェアを獲得
> - StoryGraph が5年で380万ユーザー（グローバル）を達成した成長率を参考
> - 日本市場は言語・書籍データの参入障壁がある分、シェア集中しやすい

## ビジネスの勝ち筋

### Go-to-Market 戦略

```
1. 初期ユーザー獲得（Phase 1）
   - 既存アプリへの不満層をターゲットに ASO + 読書系 SNS での露出
   - データインポート機能で乗り換えコストを極限まで下げる
   - 「ブクログ/読書メーターからの移行ガイド」コンテンツ

2. 口コミ拡散（Phase 2）
   - 読了時の SNS シェア機能で自然な拡散ループを作る
   - BookTok / 読書垢 のインフルエンサー施策
   - App Store のレビュー評価を武器にオーガニック流入を増やす

3. ネットワーク効果の発火（Phase 3）
   - ソーシャル機能で「友達も使っている」状態を作る
   - AI レコメンドの精度がユーザー数に比例して上がる → 使うほど良くなる循環
```

### マネタイズ仮説

| モデル | 内容 | タイミング |
|---|---|---|
| **基本無料** | コア機能（登録、管理、検索、リスト）はすべて無料 | ローンチ時 |
| **Shelfie Pro（サブスク）** | AI レコメンド、高度な読書統計、データエクスポート、カスタムテーマ等 | Phase 2 後半〜 |
| **アフィリエイト** | 楽天ブックス / Amazon への購入リンク経由の成果報酬 | Phase 1〜 |
| **出版社向け B2B** | 新刊プロモーション枠、読者インサイトデータ | Phase 4 |

### KPI 仮説

| フェーズ | North Star Metric | 主要 KPI |
|---|---|---|
| Phase 1 | 週次アクティブユーザー（WAU） | 新規登録数、初回登録から本3冊追加率、DAU/MAU ratio |
| Phase 2 | 月間本登録数 | リテンション率（D7, D30）、バーコードスキャン利用率 |
| Phase 3 | 月間レコメンド経由の本登録数 | ソーシャル機能利用率、シェア数、招待経由の新規ユーザー |

## リスク分析

### プロダクトリスク

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| 「モダン UI」だけでは乗り換え動機として弱い | 高 | 中 | UI だけでなく、データインポートの容易さ・検索品質・登録速度など実用面で明確に勝る体験を作る |
| 既存アプリのユーザーが乗り換えずに併用で終わる | 中 | 高 | 初回体験で「これ1つで完結する」と感じさせるオンボーディング。インポート後すぐに価値を実感できるフロー |
| AI レコメンドの実装に時間がかかり、差別化が遅れる | 高 | 中 | Phase 1-2 でコア体験の PMF を先に達成する。AI は差別化要因だが、なくても使われる基盤を先に作る |
| ビジュアル本棚訴求に寄りすぎて実用性が疎かになる | 中 | 低 | 「記録→把握→次の行動」の実用ループを最優先。見た目は手段であり目的ではない |

### 市場リスク

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| 出版市場の縮小が読書アプリ需要自体を減らす | 中 | 中 | 市場全体は縮小でも、「本を読む人が記録するかどうか」のペネトレーション率はまだ低い。パイの縮小よりシェア獲得を優先 |
| StoryGraph が日本語対応してくる | 高 | 低〜中 | 日本語書籍の検索基盤（楽天 API）と日本市場の理解が先行優位。仮に対応されても書誌データの品質で差がつく |
| 大手（Amazon, 楽天, 角川）が本気で読書アプリを作り直す | 高 | 低 | 大企業は既存事業との利害調整で動きが遅い。先行してユーザーベースとブランドを確立する |
| 読書メーターやブクログが大幅リニューアルする | 中 | 低 | 過去10年の更新頻度を見る限り可能性は低いが、常に開発速度で上回り続ける |

### 技術リスク

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| 外部 API（楽天ブックス、Google Books）の仕様変更・停止 | 高 | 低 | 自社書誌データベースの段階的構築を検討。複数 API への依存分散は実施済み |
| Flutter のメジャーアップデートによる破壊的変更 | 中 | 低 | Flutter は後方互換性を重視しており、段階的な移行が可能 |
| AI 機能のコスト（LLM API 費用）がユニットエコノミクスを圧迫 | 中 | 中 | StoryGraph のように自社モデル or 軽量モデルの活用。AI 機能はプレミアムプランに限定することでコスト制御 |

### チームリスク

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| 小規模チームで開発速度が競合に追いつかない | 中 | 中 | AI 駆動の開発ワークフロー（Claude Code 等）で生産性を最大化。機能のスコープを絞り、Phase ごとに集中 |
| ソーシャル機能の運営コスト（モデレーション等） | 中 | 中 | Phase 3 まで本格的なソーシャル機能は後回し。まずは軽量なシェア機能に留める |
