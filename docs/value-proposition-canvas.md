## なぜ人は読書記録をつけるのか — 根源的な問い

読書記録の動機は、ユーザーの読書フェーズによって根本的に異なる。Shelfie がどの動機にフォーカスするかは、プロダクトの性格を決定する。

### 読書記録の5つの動機レイヤー

```
Layer 1: 忘却防止（全ユーザー共通）
  「あの本読んだっけ？」「SNS で見た本、なんだっけ？」
  → 最も原始的で、最も普遍的な動機。全セグメントに刺さる。
  → これが Shelfie のコア。ここが弱いと何も始まらない。

Layer 2: 進捗の可視化 → 達成感 → 習慣化
  「今月3冊読んだ」「今年で20冊目」
  → 読書を"続けたい"人にとって、数字の積み上げがモチベーションになる。
  → まだ読書が習慣化していない人ほど、この仕組みが効く。

Layer 3: 学びの蓄積と深化
  「この本のあの一節が良かった」「前に読んだ本と繋がった」
  → すでに読書が習慣になっている人が、読書体験を最大化したい動機。
  → メモ、引用、気づきのストック。読書が「消費」から「資産」に変わる。

Layer 4: 次の本との出会い
  「次に何を読もう」「自分に合う本が分からない」
  → 積読リストからの選択、友人のおすすめ、AI レコメンド。
  → 読書を始めたい層にも、読書家にも刺さる。
  → Shelfie では「友人の本棚が見える」ことが最初の出会いのエンジンになる。

Layer 5: 社会的つながり
  「この本よかったよ」と伝えたい、「あの人が読んでるなら読もう」
  → プライベートな友人同士のつながりから始める。
  → オープンな SNS ではなく、信頼できる人のおすすめが流れてくる場所。
  → ネットワーク効果のエンジンであり、Shelfie のモートになる。
```

### ユーザーフェーズ別のプロダクト価値

| ユーザーフェーズ | 主な動機 | Shelfie が提供する価値 |
|---|---|---|
| **これから読書を始めたい** | Layer 4 → Layer 2 | 友達が読んでる本が見える → 「自分も読んでみよう」。1冊目を読み切った達成感の可視化 |
| **本は読むが記録していない** | Layer 1 → Layer 5 | 「2タップで記録」の手軽さで忘却防止。友達と本棚を共有する楽しさで定着 |
| **既存アプリに不満がある読書家** | Layer 1, 2, 3 | より速く、より気持ちよく記録できる体験。メモ・評価で学びを蓄積 |
| **読書体験を最大化したい層** | Layer 3 → Layer 4 | メモと気づきの蓄積、友人のおすすめから次の一冊を発見、AI レコメンド（将来） |

### Shelfie のプロダクト方針

> **Layer 1（忘却防止）と Layer 2（達成感・習慣化）を完璧にすることがすべての基盤。**
>
> Layer 1-2 は全ユーザーセグメントに共通する普遍的な動機であり、ここの体験が完璧であれば、「読書家」も「これから読みたい人」も同じプロダクトで満足できる。
>
> Layer 3（学びの深化）は Phase 1-2 で段階的に強化する。読書家の定着率を上げる鍵。
>
> **Layer 5（社会的つながり）は Phase 2 で友人同士のプライベートなつながりとして実現する。** オープンな SNS ではなく、招待制で知っている人同士がつながり、本棚を共有する形。これが Shelfie のモート（競合が真似できない防御壁）になる。友人ネットワークが定着した上で、Layer 4（次の本との出会い）は友人のおすすめ + AI レコメンド（将来）で実現する。
>
> **重要**: 「読書の習慣化」自体を Shelfie のミッションにする必要はない。Shelfie は習慣化ツールではなく、**以下の読書サイクルを回すためのインフラ**。結果として、Layer 1-2 の体験が良ければ習慣化は副産物として起きる。

### 読書サイクル — Shelfie が支えるユーザー行動の全体像

ユーザーの読書行動は、以下のサイクルとして捉えられる。Shelfie はこのサイクルのすべてのステップに自然に寄り添うプロダクトである。

```
出会う → ためる → 読む → 残す → 見返す → また出会う
  │        │       │       │        │         │
  │        │       │       │        │         └ 積読リスト・レコメンドから次の一冊へ
  │        │       │       │        └ 過去の記録を振り返り、読書パターンを把握する
  │        │       │       └ 読了を記録し、評価・メモを残す
  │        │       └ 積読から選んで読む / 読書中ステータスで進捗を意識する
  │        └ 気になった本を積読（backlog）に追加する
  └ SNS・書店・友人の本棚から本を知る
```

このサイクルには複数のエントリーポイントがある:
- **友達の本棚を見る** → 「出会う」から入る（Shelfie が最も注力するエントリーポイント）
- **SNS で本を見かける** → 「ためる」から入る
- **書店で手に取る** → 「読む」から入る
- **友人に薦められる** → 「出会う」から入る
- **過去のメモを見返す** → 「見返す」から入って次の本へ

Shelfie の役割は、このサイクルのどこから入っても自然に回り続ける状態を作ること。「記録する」は手段であり、ユーザーが本当にやりたいのは **出会って、ためて、読んで、残して、また出会う** こと。

## Customer Profile（顧客プロファイル）

### Customer Jobs（顧客が達成したいこと）

**機能的ジョブ**

- 自分が持っている本・読んだ本を把握したい
- 読みたい本を忘れないように記録しておきたい
- 今読んでいる本の進捗を管理したい
- SNS で見かけて気になった本をすぐに登録したい
- テーマやジャンルごとに本を整理したい

**感情的ジョブ**

- 「あの本なんだっけ」という不安をなくし、記録した安心感を得たい
- 読了数やステータスの変化を通じて、読書を続けるモチベーションを保ちたい
- 次に何を読むか迷ったときに、自分の積読リストから選べる状態でいたい

> **重要な区別 — 「ビジュアル本棚」と「モダン UX」は別物**
>
> **ビジュアル本棚訴求 = ユーザーニーズとしては弱い。** 「自分だけの本棚を眺めて満足」は一見キャッチーだが、実際のユーザー行動としては定着しない。ブックミー等、この軸で差別化を試みたアプリは成功しなかった。「本棚の見た目」自体はユーザーがアプリを使い続ける理由にならない。ユーザーの本質的な欲求は「出会った本をためて、読んで、残して、見返せる」という読書サイクルが途切れずに回ること。
>
> **モダン UX = 初期の最大の差別化要因であり、ブランドそのもの。** 一方で、パフォーマンス（画面遷移の速さ、滑らかなアニメーション）、洗練されたビジュアルデザイン（ダークモード、余白の使い方、タイポグラフィ）、直感的なインタラクション（ジェスチャー操作、ハプティックフィードバック）は、実用機能と一体になってアプリの品質を形成する。これは「見た目がきれい」という装飾の話ではなく、**「使っていて気持ちいい」=「もう古いアプリには戻れない」** という体験のスイッチングコストを作る戦略。
>
> 国内の競合アプリが軒並み UI/UX の刷新に失敗している現状で、**モダン UX は Phase 1 における最大の乗り換え動機**になる。これは「キレイな本棚を作りたい」というニッチな欲求とは根本的に異なり、「イケてるアプリを使いたい」「もっさりしたアプリはストレス」というはるかに広いユーザー心理に刺さる。
>
> つまり：
> - ~~ビジュアル本棚~~ → プロダクトの存在意義（Why）にはならない
> - **モダン UX** → 初期の差別化（How）として極めて有効。ブランド認知にも直結する

**社会的ジョブ**

- 友達が何を読んでいるか知りたい、自分の本棚も友達に見せたい
- 読んだ本やおすすめを信頼できる友人と共有したい
- 友達のおすすめから次に読む本を見つけたい

### Pains（ペイン）

**本の管理に関するペイン**

- 「あの本読んだっけ？」と購入済み・既読の本を忘れてしまう
- 積読が増えるが、何が積まれているか把握できない
- SNS で気になった本を見つけても、後で思い出せない

**既存アプリに対するペイン**

- **UI/UX が古い・もっさりしている** — 主要アプリ（ブクログ・読書メーター）は10年以上前からのサービスで、画面遷移が遅い、デザインが時代遅れ
- **機能過多で取っつきにくい** — コミュニティ・談話室・レビューSNS 等の機能が多すぎて、単に本を記録したいだけの人には重い
- **AI・最新技術への対応が遅い** — 海外では StoryGraph が AI レコメンデーション・ムードベース検索を実装している一方、国内主要アプリは AI 活用がほぼゼロ
- **開発が停滞気味** — 長期運営の国内アプリは大きな機能更新が少なく、モダンな UX トレンド（ダークモード、アニメーション、ジェスチャー操作等）に追従していない
- **プラットフォーム制約** — ビブリアは iOS 専用、読書メーターはデータエクスポート不可など、各アプリに構造的制約がある

---

### Gains（ゲイン）

- すべての蔵書をいつでもどこでも確認できる
- バーコードをかざすだけで本を登録できる手軽さ
- 読書状況（積読 / 読書中 / 読了 / 読まない）が一目でわかる
- お気に入りリストで本を自由にグルーピングできる
- 読了した本に評価やメモを残せる
- 使っていて心地よい、モダンで洗練された操作体験
- 友達が何を読んでいるか見える、自分の本棚も共有できる
- 信頼できる友人のおすすめから次の本に出会える

## 既存の代替手段と課題

### 国内主要アプリ

| アプリ | ユーザー規模 | 強み | 課題・ペイン |
|---|---|---|---|
| **ブクログ** | 100万人+（ブックオフグループ） | 管理機能が充実、タグ分類、新刊ニュース、CSV エクスポート | UI が古い、SNS 機能弱い、ソート機能不足、Web サービス発の設計 |
| **読書メーター** | 国内最大級（角川/ドワンゴグループ） | コミュニティが活発、1,800万件超のレビュー蓄積 | 画面遷移が遅い、5段階評価なし、データエクスポート不可、並べ替え不可 |
| **ビブリア** | - | シンプルで見やすい、ソート充実 | iOS 専用、カテゴリ分類不可、SNS 機能なし |
| **Readee**（楽天） | - | 楽天購入履歴と自動連携 | 楽天エコシステム依存、機能が限定的 |
| **ブックミー** | - | ビジュアル本棚に特化 | 「本棚を眺める」訴求だけでは定着せず、実用性が不足 |

**共通する構造的課題**:
- いずれも2010年代前半までに確立されたサービスで、UI/UX のモダナイズが遅れている
- AI・機械学習の活用がほぼない（レコメンデーション、自動分類、読書インサイト等）
- 開発リソースが限られ、大きな機能進化が停滞している

### 海外アプリ

| アプリ | ユーザー規模 | 強み | 日本市場での課題 |
|---|---|---|---|
| **Goodreads**（Amazon） | 約1.5億人 | 世界最大の読書 SNS、圧倒的な書誌データ | 日本語書籍のカバレッジが低い、UI が古い、開発停滞 |
| **StoryGraph** | 約380万人 | AI レコメンデーション、ムード検索、詳細な統計 | 日本語未対応、日本の書籍データが不足 |
| **Literal Club** | - | モダン UI、人ベースのレコメンド、オープン API | 日本語未対応、日本のユーザーベースなし |

**海外アプリの共通課題**: 技術的には先進的だが、日本語書籍・楽天/Amazon.co.jp との連携がなく、日本市場では実用的でない

## 3C 分析

### Customer（市場・顧客）

**市場環境**

- 日本の出版市場（紙＋電子）は2025年で約1兆5,462億円（前年比 ▲1.6%、4年連続マイナス）
- 紙の出版は9,647億円（▲4.1%）で縮小傾向だが、Z世代の73%は紙の本を好む
- 電子出版は5,815億円（+2.7%）で伸長するも、成長率は鈍化
- 電子書籍市場は2029年度に8,000億円弱まで成長見込み（年平均成長率 3.1%）

**ユーザー像**

| セグメント | 特徴 | 読書頻度 | 現在の代替手段 |
|---|---|---|---|
| 既存アプリ不満層 | ブクログ/読書メーターを使っているが UI に不満 | 月1冊以上 | ブクログ、読書メーター |
| 記録していない読書層 | 本は読むが記録していない、管理が面倒 | 月1〜3冊 | なし、メモアプリ、SNS投稿 |
| 若年ライト読書層（10代〜20代） | SNS で本を発見、年に数冊読む | 年数冊〜月1冊 | SNS のブックマーク、なし |
| 読書始めたい層 | 読書に興味はあるが何を読むか分からない | ほぼゼロ | SNS、書店 |

**Z 世代の読書行動データ**（マイナビ 2025年調査 / MERY調査）

- Z 世代社会人の38%が読書習慣あり（62%はなし）
- 読む人の73%は紙の本を好み、購入場所は書店が7割超
- 読みたい本を見つけるきっかけ: **SNS が46.2%で最多**
- 月あたりの読書量ボリュームゾーン: 1〜3冊
- 月あたりの書籍費: 500〜3,000円
- 人気ジャンル: 文学・小説 63.5%、経済・ビジネス 34.3%

### Competitor（競合）

**国内競合の構造的弱点**

| 競合 | 運営母体 | 開発体制 | AI 対応 | 弱点 |
|---|---|---|---|---|
| ブクログ | ブックオフグループ | 小規模・維持運用中心 | なし | 書店流通企業が母体、テック投資の優先度が低い |
| 読書メーター | 角川/ドワンゴ（トリスタ） | 中規模 | なし | 大企業傘下だが本サービスへの注力は限定的 |
| ビブリア | 個人開発 | 1名 | なし | iOS のみ、開発リソースに限界 |
| Goodreads | Amazon | ほぼ凍結状態 | なし | Amazon の戦略的優先度が低く、長年放置されている |

**海外新興の動き**

- StoryGraph: AI レコメンデーションで急成長（380万ユーザー）。ただし日本語圏は未対応
- Literal Club: モダン UI で注目されるも、スケールはまだ小さい
- Hardcover / Basmo: ニッチプレイヤー、日本進出の動きなし

### Company（自社）

**強み**

- モダンな技術スタック（Flutter, GraphQL, PostgreSQL）で高速に機能開発できる
- 日本語書籍に強い検索基盤（楽天ブックス + Google Books 横断検索）
- iOS / Android 同時対応のクロスプラットフォーム開発
- AI 機能を追加できる拡張性のあるアーキテクチャ
- 意思決定が速い小規模チーム

**弱み**

- ユーザーベースがゼロからのスタート（ネットワーク効果がない）
- ブランド認知なし
- 書誌データは外部 API 依存（楽天・Google）
- マネタイズモデルが未確定

### Pain Relievers（ペインの解消）

| ペイン | 解消方法 |
|---|---|
| 既読・未読を忘れる | 4つの読書ステータスで管理。本棚画面で一目で把握 |
| 積読が把握できない | 「積読（backlog）」セクションに集約表示 |
| SNS で見た本を忘れる | バーコードスキャンまたは検索で即登録。最短2タップで本棚に追加 |
| 既存アプリの UI が古い・遅い | Flutter × Material 3 でネイティブ級のパフォーマンスとモダン UI を実現。ダークモード標準、滑らかなアニメーション |

## Fit（フィット分析）

### コアバリュー

> **「あなたの本棚を、手のひらに。友達の本棚も。」**
>
> 既存の読書アプリが抱える「古い・重い・多すぎる」を解消し、
> 気持ちいい読書管理と友達との本棚共有で、読書をもっと楽しくする。
>
> - **Why（存在意義）**: 気になった本を即記録し、友達と読書体験を共有できる場所
> - **How（差別化）**: 国内アプリで唯一の「モダン UX × 日本語書籍 × プライベートな友人つながり」。使い心地と友人ネットワークが競合との壁になる
> - **What（機能）**: バーコードスキャン、ステータス管理、ブックリスト、評価・メモ、友達の本棚閲覧 → AI レコメンド（将来）

### ターゲットユーザー

**初期フォーカス（ローンチ〜PMF）**

- 年間5冊以上本を読み、蔵書を記録・管理したいと感じている人
- 既存の読書アプリ（ブクログ・読書メーター等）を使ったが、UI の古さや使いにくさに不満を感じている人
- シンプルで洗練されたツールを好む人

**将来ビジョン**

- 10代〜20代を含む、本を読む（または読みたい）すべての人の must-have
- 「読書家」を自称しない層 — SNS で話題の本が気になる、年に数冊は読む、これから読書を始めたい人
- Z 世代の読書行動（SNS で本を発見、紙の本を好む）にフィットするプロダクト

**広がりの仮説**: 「狭く深く」→「広く深く」の段階的拡大

```
Phase 1: 既存読書アプリに不満を持つ管理意識の高い読書家
         → シンプル × モダン UI で乗り換え需要を獲得
Phase 2: 友達と本の話がしたい読書層
         → 招待制の友人つながりで「友達の本棚が見える」体験を提供
         → 友達を招待する成長ループで自然にユーザーが広がる
Phase 3: 本を読みたいと思っている・SNS で本に触れている若年層
         → 本棚の公開オプション（LINE 型 → Instagram 型への拡張）
         → AI レコメンデーションで「次の一冊」との出会いを提供
```

### 競合との差別化ポイント

| 観点 | 国内既存アプリ | 海外先進アプリ | **Shelfie** |
|---|---|---|---|
| UI/UX | レガシー、画面遷移が遅い | モダンだが日本語非対応 | **モダン × 日本語ネイティブ** |
| 日本語書籍 | 強い（楽天連携等） | 弱い | **楽天 + Google 横断検索** |
| ソーシャル | 読書メーターはオープンコミュニティ。ビブリアはなし | Goodreads はオープン SNS | **プライベートな友人つながり（差別化軸）** |
| AI 活用 | ほぼなし | StoryGraph がリード | **将来の差別化領域** |

### Shelfie が勝てる理由（エビデンス付き）

**1. 国内既存アプリへの不満が顕在化している**

App Store レビューの実際の声:

> **読書メーター** — レビュー感情分析: 不満52%、喜び42%、怒り6%（不満が過半数）
> - 「バーコードを読み込んでもすぐ反映されず、一旦アプリを閉じて立ち上げないと反映されない」
> - 「操作の説明等がなく、感覚で操作出来るものではなかった」
> - 「書籍タイトルで検索しても関係ない本しか出てこない」
> - 「感想を入力して『登録する』ボタンで、入力した感想が全部消えた」
> - 「運営に申し出てもなしの礫。改善もない」
>
> **ブクログ**
> - 「本棚内検索が出来なくなった為、所持品管理として機能しなくなりました」（星1）
> - 「検索後に元の本棚に戻ることが出来ない...わざわざアプリを立ち上げ直さなければならない」（星3）
> - 「本棚内を更新すると、並び替えが初期設定に戻ってしまう」（星3）
> - 「フィルタが一つしかかからない改悪」（星1）

出典: [ブクログ App Store](https://apps.apple.com/jp/app/%E8%AA%AD%E6%9B%B8%E7%AE%A1%E7%90%86%E3%83%96%E3%82%AF%E3%83%AD%E3%82%B0-%E6%9C%AC%E6%A3%9A-%E8%AA%AD%E6%9B%B8%E8%A8%98%E9%8C%B2/id359066582) / [読書メーター App Store](https://apps.apple.com/jp/app/%E8%AA%AD%E6%9B%B8%E3%83%A1%E3%83%BC%E3%82%BF%E3%83%BC/id509979590) / [app-ranking.net レビュー分析](https://www.app-ranking.net/id/509979590)

**結論**: UI の古さだけでなく、「検索が壊れている」「データが消える」「運営が改善しない」という品質・信頼の問題。不満は存在するが、ユーザーは「他に選択肢がない」から使い続けている状態。

---

**2. 海外の先進アプリが日本市場に入れていない**

- StoryGraph / Goodreads ともに**日本語の UI を提供していない**（英語のみ）
- 日本語ブログの比較記事でも「日本語の本は Goodreads を使い、英語の本は StoryGraph で使うなど、使い分けるのもいいアイデア」と紹介されている（=StoryGraph では日本語書籍を管理できない前提）
- HiNative の Q&A では「日本人は Goodreads ではなく、Amazon Japan / Yahoo / 楽天 でレビューを読む」との回答
- StoryGraph の書誌データは主に英語圏の ISBN データベースに依存しており、楽天ブックス・日本の出版社のデータとは非連携

出典: [StoryGraph vs Goodreads 比較（Currently Listening）](https://www.currentlylistening.com/blog/books-storygraph-vs-goodreads) / [HiNative: Where do Japanese people review books?](https://hinative.com/questions/24472014)

**結論**: 技術的にモダンな海外アプリは存在するが、日本語書籍を実用的に検索・登録できるものは現時点でゼロ。楽天ブックス API や日本の ISBN 体系への対応が参入障壁になっている。

---

**3. 「モダン UI × 日本語書籍 × プライベートソーシャル」の交差点が空白**

- **ブクログ / 読書メーター**: 日本語書籍に強いが、ソーシャルはオープンなコミュニティ型。友人同士のプライベートなつながりは提供していない
- **StoryGraph**: AI レコメンド有、モダン UI だが、日本語未対応。ソーシャルはフォロー型
- **Goodreads**: 世界最大の読書 SNS だが、オープンなレビューコミュニティ。日本語書籍に弱い
- **ビブリア**: シンプルで優秀だが、ソーシャル機能なし。iOS 専用
- 2025年版の読書記録アプリランキング（マイベスト）でも、「友人と本棚を共有する」機能を持つ読書管理アプリは**1つもランクインしていない**

出典: [読書記録アプリのおすすめ人気ランキング 2025（マイベスト）](https://my-best.com/6121)

**結論**: 日本語で使えて、モダンな UX を持ち、友人同士でプライベートに本棚を共有できる読書管理アプリは 2026年2月時点で確認できない。既存のソーシャル読書アプリはすべてオープンコミュニティ型であり、「信頼できる友人のおすすめ」という最も強い購買動機に応えるプロダクトは空白。

---

4. **技術的に正しいスタックを選択済み** — Flutter, GraphQL, モダン API 設計で継続的な進化が可能

## ロードマップ仮説

### Phase 1: コア体験の磨き込み（現在〜PMF）

**目標**: 既存アプリからの乗り換えユーザーが「これでいい」ではなく「これがいい」と感じる体験

| 注力領域 | 具体的な施策 |
|---|---|
| 登録体験の最適化 | バーコードスキャンの精度・速度向上、検索結果の品質改善 |

| データ移行 | ブクログ・読書メーターからの CSV / データインポート機能 |
| 安定性・パフォーマンス | オフライン対応強化、起動速度・画面遷移の最適化 |

**目標**: 友達と本棚を共有する体験で定着率を上げ、招待ループでユーザーを広げる

| 注力領域 | 具体的な施策 |
|---|---|
| 友達とつながる | 招待リンク（LINE 等）、ユーザー検索（ID / 名前）、QR コード交換 |
| 友達の本棚閲覧 | 友達が最近追加した本・読了した本が見える |

| ウィジェット | ホーム画面に「今読んでいる本」「次に読む本」を表示 |

> **設計方針**: 本棚はデフォルト非公開。つながった友達にだけ見える。おすすめユーザーやランキングは設けない（LINE 型）。グループ機能は作らない（グループは過疎って終わるため）。

### Phase 3: オープン化と発見（グロース〜スケール）

**目標**: クローズドからオープンへ拡張し、10代〜20代の「読書家じゃない層」の must-have になる

| 注力領域 | 具体的な施策 |

| 友達のおすすめから積読追加 | 友達が読了した本をワンタップで自分の積読に追加 |
| 共有ブックリスト | 友達と共同でブックリストを作成・管理 |
| AI レコメンデーション | 読書履歴・評価に基づく「あなたへのおすすめ」、友人の読書傾向を加味した提案 |
| SNS 連携 | TikTok/Instagram で話題の本のトレンド表示、SNS 投稿からの本登録 |

### Phase 4: プラットフォーム化（スケール後）

| 注力領域 | 具体的な施策 |
|---|---|
| 出版社・書店連携 | 新刊情報、書店在庫検索、購入導線 |

### 課金機能の実装タイミング

**結論: Phase 1 後半（PMF 確認後すぐ）に導入すべき。Phase 2 まで待つのは遅い。**

```
❌ ローンチ直後（Phase 1 前半）
   まだユーザーが価値を実感する前。課金ウォールは離脱を招くだけ。

✅ Phase 1 後半（PMF の兆候が見えたタイミング）
   コアユーザーが「もう手放せない」と感じ始めた段階。
   ここで Shelfie Pro を導入し、早期ユーザーをプレミアム会員に転換する。

❌ Phase 2 以降まで待つ
   全機能無料に慣れたユーザーに後から課金を求めるのは極めて難しい。
   StoryGraph がこのパターンで、380万ユーザーに対して月間収益$20K（≒¥300万）に留まっている。
```

**Phase 1 後半で導入する理由**

1. **「無料で全部使える」を既成事実にしない**
   - ユーザーが「無料で当然」と認識する前に、Free / Pro の境界線を引く
   - 初期から「Pro はある」という認知を作ることで、将来の課金抵抗を下げる

2. **RevenueCat のデータが示す教訓**
   - アプリのインストール→サブスク課金率の業界平均は約2%
   - しかし無料トライアル提供時のコンバージョンは10〜29%に跳ね上がる
   - 早期に無料トライアル付きの Pro プランを提示することで、課金率を大きく改善できる

3. **ユニットエコノミクスの早期検証**
   - LTV（顧客生涯価値）と CAC（顧客獲得コスト）の関係を Phase 1 で把握する
   - 「¥480/月で課金する人が何%いるか」は、仮説ではなく実データで検証すべき

**Free / Pro の線引き案**

| | Free | Pro（¥480/月） |
|---|---|---|
| 本の登録・検索 | 無制限 | 無制限 |
| バーコードスキャン | 無制限 | 無制限 |
| 読書ステータス管理 | 無制限 | 無制限 |
| ブックリスト | 3つまで | 無制限 |
| 読書統計 | 基本（冊数のみ） | 詳細（ジャンル分布、ペース、推移） |
| データエクスポート | - | CSV / JSON |
| AI レコメンド（将来） | - | 利用可能 |
| カスタムテーマ | - | 利用可能 |

> **設計原則**: Free でも読書サイクル（出会う→ためる→読む→残す→見返す→また出会う）は完全に回せる。Pro は「分析・深化・発見」の上位レイヤー（詳細統計、AI レコメンド等）を提供する。Free ユーザーが「もっと知りたい」と思ったときに自然に Pro を検討する導線。

**TAM（Total Addressable Market）— 最大市場規模**

> **約4,900万人**（5,400万人 × 90%）
>
> 仮にこの全員が月額480円のサブスクに加入した場合の理論上限:
> 4,900万人 × ¥5,760/年 = **約2,822億円/年**

※これは理論上限であり、到達可能な数字ではない。市場のポテンシャルを示す指標。

**SAM（Serviceable Addressable Market）— 提供可能市場**

読書管理アプリに関心を持ちうるユーザー層（定期的に読書し、記録に関心がある or 既存アプリを利用中）

> **約800〜1,200万人**
>
> 算出根拠:
> - 月1冊以上読む4,040万人のうち、記録ニーズがある層を20〜30%と推定
> - 既存アプリユーザー500〜800万人 + 潜在的な新規層
>
> サブスク（課金率5%想定）+ アフィリエイト: **約35〜53億円/年**
>
> | 収益源 | 計算 | 金額 |
> |---|---|---|
> | サブスク | 1,000万人 × 5% × ¥5,760/年 | 28.8億円 |
> | アフィリエイト | 1,000万人 × 10% 購入率 × ¥1,200/件 × 3% 報酬率 × 年6回 | 2.2億円 |

> **Phase 3 到達時（3〜5年後）: MAU 50〜200万人**

> - 国内読書管理アプリ市場で10〜20%のシェアを獲得

## ビジネスの勝ち筋

### Go-to-Market 戦略

```
1. 初期ユーザー獲得（Phase 1）
   - 既存アプリへの不満層をターゲットに ASO + 読書系 SNS での露出
   - データインポート機能で乗り換えコストを極限まで下げる
   - 読了カードの画像シェアでブランド認知の種まき

2. 友人招待ループ（Phase 2）
   - 友達とつながる機能リリースで「友達を誘う理由」が生まれる
   - 「友達が何読んでるか見れるよ」が最も強い招待文句
   - 1人が3人招待 → その3人がさらに招待、のウイルスループ
   - BookTok / 読書垢 のインフルエンサー施策も並行

3. ネットワーク効果の発火（Phase 3）
   - 友人グループ単位でのロックインが完成
   - 本棚公開オプションでオーガニック流入を増やす
   - AI レコメンドの精度がユーザー数に比例して上がる → 使うほど良くなる循環
```

### マネタイズ仮説

| モデル | 内容 | タイミング |
|---|---|---|
| **基本無料** | コア機能（登録、管理、検索、リスト）はすべて無料 | ローンチ時 |
| **Shelfie Pro（サブスク）** | 高度な読書統計、AI レコメンド、データエクスポート、カスタムテーマ等 | PMF 確認後〜 |
| **アフィリエイト** | 楽天ブックス / Amazon への購入リンク経由の成果報酬。友人おすすめ経由の購入は特にコンバージョンが高い | Phase 1〜 |
| **出版社向け B2B** | 新刊プロモーション枠、読者インサイトデータ | Phase 4 |

### KPI 仮説

| フェーズ | North Star Metric | 主要 KPI |
|---|---|---|
| Phase 1 | 週次アクティブユーザー（WAU） | 新規登録数、初回登録から本3冊追加率、DAU/MAU ratio |
| Phase 2 | 招待経由の新規ユーザー数 | 友達追加率、友達の本棚閲覧頻度、リテンション率（D7, D30） |
| Phase 3 | 友人おすすめ経由の本登録数 | 本棚公開率、フォロー数、AI レコメンド経由の本登録数 |

## リスク分析

### プロダクトリスク

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| 「モダン UI」だけでは乗り換え動機として弱い | 高 | 中 | UI だけでなく、データインポートの容易さ・検索品質・登録速度など実用面で明確に勝る体験を作る |
| 既存アプリのユーザーが乗り換えずに併用で終わる | 中 | 高 | 初回体験で「これ1つで完結する」と感じさせるオンボーディング。インポート後すぐに価値を実感できるフロー |
| 友人つながり機能を入れても友達を招待してくれない | 高 | 中 | 招待の動機が自然に生まれる設計（「あの人が何読んでるか見たい」）。招待しなくても個人の読書管理として成立する土台を維持 |
| ソーシャル機能の開発で個人管理の磨き込みが遅れる | 中 | 中 | Phase 1 でコア体験の PMF を先に達成。ソーシャルは Phase 2 で段階的に投入 |

### 市場リスク

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| 出版市場の縮小が読書アプリ需要自体を減らす | 中 | 中 | 市場全体は縮小でも、「本を読む人が記録するかどうか」のペネトレーション率はまだ低い。パイの縮小よりシェア獲得を優先 |
| StoryGraph が日本語対応してくる | 高 | 低〜中 | 日本語書籍の検索基盤（楽天 API）と日本市場の理解が先行優位。仮に対応されても書誌データの品質で差がつく |
| 大手（Amazon, 楽天, 角川）が本気で読書アプリを作り直す | 高 | 低 | 大企業は既存事業との利害調整で動きが遅い。先行してユーザーベースとブランドを確立する |
| 読書メーターやブクログが大幅リニューアルする | 中 | 低 | 過去10年の更新頻度を見る限り可能性は低いが、常に開発速度で上回り続ける |

### 技術リスク

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| 外部 API（楽天ブックス、Google Books）の仕様変更・停止 | 高 | 低 | 自社書誌データベースの段階的構築を検討。複数 API への依存分散は実施済み |
| Flutter のメジャーアップデートによる破壊的変更 | 中 | 低 | Flutter は後方互換性を重視しており、段階的な移行が可能 |
| AI 機能のコスト（LLM API 費用）がユニットエコノミクスを圧迫 | 中 | 中 | StoryGraph のように自社モデル or 軽量モデルの活用。AI 機能はプレミアムプランに限定することでコスト制御 |

### チームリスク

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| 小規模チームで開発速度が競合に追いつかない | 中 | 中 | AI 駆動の開発ワークフロー（Claude Code 等）で生産性を最大化。機能のスコープを絞り、Phase ごとに集中 |
| ソーシャル機能の運営コスト（モデレーション等） | 低 | 低 | 招待制のクローズドなつながりのため、モデレーションコストは最小。オープン化（Phase 3）時に改めて検討 |
