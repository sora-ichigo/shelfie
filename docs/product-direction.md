# Shelfie プロダクト方向性

> **ステータス**: 意思決定済み
> **最終更新**: 2026-02-08

---

## 結論

**「気持ちいい読書管理 + 友達と本棚を共有する」アプリ**

| 要素 | 内容 |
|---|---|
| 土台 | シンプルで気持ちいい読書管理（継続して磨く） |
| モート | 招待制の友人ネットワーク → 小集団のスイッチングコスト |
| 成長 | 友達を招待 → 本棚が見える → 自然な口コミ |
| マネタイズ | 定着してから設計（Pro / アフィリエイト） |
| AI | ソーシャルの先に載せる |

---

## 議論の経緯

### 4つの方向性を検討

| 方向性 | 概要 | 評価 |
|---|---|---|
| シンプルな読書管理（post ビブリア） | 登録・管理の UX を極限まで磨く | 市場の天井が低い。ビジネスとしてスケールしにくい |
| AI 読書管理 | AIレコメンド、ムード検索、読書パターン分析 | 空白市場だが、冷たいスタート問題。ユーザーデータがないと価値が出ない |
| 読書 SNS（post reads、ブックミー） | フォロー/フォロワー、タイムライン、レビュー | ネットワーク効果は最強のモートだが、ゼロからの構築は極めて困難 |
| 書籍情報サービス（post ブクログ、読書メーター） | レビュー、ランキング、新刊情報 | 既存サービスのデータ蓄積に勝てない |

### 第5の方向性: プライベートな読書つながり

4つの方向性を検討する中で、**オープンな SNS ではなく、プライベートな友人同士のつながり**に焦点を当てる方向が浮上した。

着想: 「友達と本の話がしたい」「友達が何を読んでいるか知りたい」というシンプルな欲求。

Strava モデルとの類似性:

| Strava | Shelfie |
|---|---|
| 個人のランニング記録 | 個人の読書記録 |
| 友達のアクティビティが見える | 友達の読了・積読が見える |
| 「あの人も走ってる」→ 自分も走る | 「あの人これ読んだんだ」→ 自分も読む |
| 小さなグループの中で刺激し合う | 小さなグループの中でおすすめし合う |

### つながり方の設計

3つの選択肢を検討した:

| 方式 | 特徴 |
|---|---|
| A. 招待制（LINE 的） | クローズド。知ってる人だけ |
| B. フォロー制（Instagram 的） | オープン。気軽にフォロー |
| C. グループ制（Discord 的） | グループ単位で招待 |

**決定: LINE 型（1対1の友達関係）から始めて、後から Instagram 型に拡張する。グループ制は採用しない。**

A を選んだ理由:

- 「友達と本の話がしたい」という原体験は、すでに知っている人との関係
- フォロー制でいきなり始めると「誰もいない SNS」になるリスクがある
- クローズドからオープンへの拡張は自然にできる（逆は難しい）
- 小集団（3〜5人）で価値が成立する → ニワトリと卵問題を回避

C（グループ制）を採用しない理由:

- **グループは過疎って終わる**。LINE グループも Slack チャンネルも大体そうなる。読書のペースは人それぞれで、グループ全体のアクティビティを維持するのは現実的でない
- グループのセットアップコストが高い（作成 → 招待の2ステップ）
- 「あの人の本棚が見たい」は1対1の関心であり、グループ単位の体験ではない
- グループ運営の面倒さ（誰が管理する、誰を入れる等）が発生する

ただし、1対1の友達関係が土台にあれば、その上に「読書グループ」的な機能を後から載せることは可能。排他的な選択ではなく順番の問題。とはいえ優先度は低い。

### つながり方の具体的な経路

純粋な招待制だけでは「アプリを入れてよ」のハードルが高いため、複数の経路を用意する:

| 経路 | 用途 |
|---|---|
| 招待リンク（LINE 等で送る） | まだアプリを使ってない友達を誘う |
| ユーザー検索（ID / 名前） | すでにアプリにいる友達を見つける |
| QR コード | 対面で交換 |

ただし、おすすめユーザーやランキングは設けない。知らない人とは繋がらない設計。本棚はデフォルト非公開で、繋がった人にだけ見える。

## モートの構造

Shelfie の現時点（v1.1.0）には明確なモートがない。この方向性で構築されるモートは**小集団のネットワーク効果**。

- 友人グループが Shelfie に乗ったら、一人だけ別アプリに移れない
- オープン SNS の大規模ネットワーク効果より地味だが、解約しにくさは強い
- ユーザーの本棚データの蓄積がスイッチングコストを高める

現時点の最大の武器はモートではなく**スピード**。国内既存アプリは開発停滞、海外アプリは日本語未対応。この空白が埋まる前にポジションを取り、友人ネットワークでモートを構築する。

## マネタイズ

ソーシャル機能自体はマネタイズしにくい（有料にするとネットワーク効果が死ぬ）。ソーシャルは**成長エンジン**であって**収益エンジン**ではない。

定着後のマネタイズ候補:

| 収益源 | 内容 |
|---|---|
| アフィリエイト | 友人のおすすめ経由の購入はコンバージョンが高い |
| Shelfie Pro（サブスク） | 読書統計、AI レコメンド、無制限リスト等 |
| グループ体験 | 読書チャレンジ、共有リスト等のプレミアム機能 |

**方針: まず使われるものを作る。マネタイズは定着してから設計する。**

## ロードマップ

### ソーシャル機能の段階的拡張

```
Step 1: 友達とつながる（招待リンク、ユーザー検索、QR コード）
Step 2: 友達の本棚が見える、最近読んだ本がわかる
Step 3: 「気になる」リアクション
Step 4: 友達のおすすめから積読に追加
Step 5: 共有ブックリスト
Step 6: 読書チャレンジ
```

### クローズド → オープンへの拡張パス

```
Phase 1: LINE 型（クローズド）
  知ってる人とだけつながる。本棚はデフォルト非公開。

Phase 2: Instagram 型を追加（オープン）
  本棚を「公開」にできるオプション追加。
  公開ユーザーは検索・フォロー可能に。
```

クローズドで先にコンテンツの質が担保される。ユーザーの本棚が充実した状態でオープン化することで、新規ユーザーが入ってきたときの体験が良くなる。

公開に切り替わる自然なトリガー:

- 本棚が 30 冊くらい溜まって見栄えがする
- 友達から「本棚見たい」と言われる
- 読了シェア画像を SNS に投稿している（もう公開しているのと同じ）

### 全体のレイヤー構造

```
土台: シンプルで気持ちいい読書管理（今ここ）
  ↓
モート: 招待制の友人ネットワーク
  ↓
拡張: オープンなフォロー制
  ↓
収益: Pro サブスク / アフィリエイト / AI レコメンド
```
